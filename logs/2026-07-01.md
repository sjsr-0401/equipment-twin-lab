## 소프트웨어 아키텍처 설명

```text
Scenario JSON
    ↓
ScenarioStep
    ↓
ScenarioRunner
    ↓
MotionAxis
    ↓
Scenario expectation
    ↓
CLI Markdown report
```

`ScenarioStepAction`은 “JSON에서 사용할 수 있는 명령 이름 목록”이다.

`ScenarioStep`은 그 명령에 필요한 필드를 검증한다. 예를 들어 `StartMotionMove`는 `axis`, `targetPosition`, `durationMilliseconds`가 필요하다.

`ScenarioRunner`는 실제로 `MotionAxis`를 찾아서 명령을 실행한다. 축은 이름별로 보관되므로 `X`축에 Servo On을 한 뒤 다음 step에서 같은 `X`축을 Home/Move할 수 있다.

CLI는 계산하지 않는다. Core가 실행한 축 상태를 읽어서 리포트에 표시한다.

# Goal 024: Fault Expected-Failure Report

## 오늘 추가로 한 일

- PR #21을 Ready로 전환하고 squash merge했다.
- 로컬 `main`을 PR #21 병합 커밋으로 최신화했다.

- CLI `template run`에 `--expect-execution-failure` 옵션을 추가했다.
- expected fault 실행 시 `Execution: FAIL`이 기대값이면 exit code 0을 반환하게 했다.
- 기존 fault 실행은 옵션이 없으면 exit code 1을 유지하게 했다.
- `--expect-execution-failure`는 반드시 `--fault`와 같이 쓰도록 제한했다.
- 콘솔과 Markdown report에 expected execution/expectation 결과를 추가했다.
- CI와 Visual Studio launch profile에 expected fault 실행을 추가했다.
- README, architecture, validation, Visual Studio 문서를 갱신했다.
- Draft PR #22를 만들었다.
- GitHub Actions push/pull_request CI 성공을 확인했다.

## 왜 필요한가

fault 주입 테스트는 일부러 장비 실행을 실패시킨다.

예를 들어 X축 Timeout은 `Execution: FAIL`이 맞다.

하지만 CI는 exit code 1을 실패로 보기 때문에, fault 재현 테스트를 안전하게 돌리려면 “이번 실패는 기대값”이라는 표시가 필요하다.

```text
Execution: FAIL
Expected execution: FAIL
Execution expectation: MET
```

- `scripts/Invoke-UnitySmokeTest.ps1` 실행은 Unity까지 도달했지만 현재 상태에서는 `No valid Unity Editor license found`로 종료됐다.

Unity Hub에서 로그인/라이선스 활성화 후 실제 화면을 열어 스크린샷/데모 체크리스트를 만든다.
