## 왜 필요한가

Goal 014의 `MotionAxis`는 축 자체 모델이었다.

이번 작업은 그 축을 사용자가 고를 수 있는 시나리오 파일로 끌어올린 것이다.

즉, 아래처럼 발전했다.

```text
테스트 코드 안의 MotionAxis
    ↓
JSON으로 실행 가능한 MotionAxis
    ↓
나중에 Unity UI / Equipment Template / Product Recipe가 재사용 가능한 구조
```

## 소프트웨어 아키텍처 설명

```text
Scenario JSON
    ↓
ScenarioStep
    ↓
ScenarioRunner
    ↓
MotionAxis
    ↓
Scenario expectation
    ↓
CLI Markdown report
```

`ScenarioStepAction`은 “JSON에서 사용할 수 있는 명령 이름 목록”이다.

`ScenarioStep`은 그 명령에 필요한 필드를 검증한다. 예를 들어 `StartMotionMove`는 `axis`, `targetPosition`, `durationMilliseconds`가 필요하다.

`ScenarioRunner`는 실제로 `MotionAxis`를 찾아서 명령을 실행한다. 축은 이름별로 보관되므로 `X`축에 Servo On을 한 뒤 다음 step에서 같은 `X`축을 Home/Move할 수 있다.

CLI는 계산하지 않는다. Core가 실행한 축 상태를 읽어서 리포트에 표시한다.

## 사용자가 이해해야 할 개념

이번 작업의 핵심은 “코드로만 테스트하던 동작을 데이터로 실행하게 만든 것”이다.

장비 시뮬레이터가 커지면 사용자가 매번 C# 코드를 고쳐서 테스트할 수 없다. JSON 같은 데이터 파일로 장비 동작을 표현해야 Unity UI, CLI, 자동화 루프가 같은 시나리오를 돌릴 수 있다.
