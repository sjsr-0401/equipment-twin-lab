## 사용자가 이해해야 할 것

이번 작업의 핵심은 코드를 만든 것이 아니라, Codex와 Claude Cowork가 같은 기준으로 일할 수 있게 “기억 장치”를 만든 것이다.

에이전트는 대화가 끊기면 맥락을 잊을 수 있지만, 파일에 남긴 상태와 결정사항은 다음 작업에서 다시 읽을 수 있다.

추가로 첫 코드 MVP를 만들었다.

장비 상태머신은 장비가 현재 어떤 단계에 있는지 관리하는 기본 구조다. 실제 장비에서는 PLC 신호, 모션 완료, 센서 입력, 카메라 검사 완료 같은 이벤트가 상태 전이를 일으킨다.

가상 IO 모델은 실제 PLC 없이 센서 입력과 출력 명령을 분리해서 테스트하는 구조다.

Input은 센서/PLC가 바꾸고 장비 SW가 읽는 값이다. Output은 장비 SW가 쓰고 밸브/램프/부저 같은 장치가 반응하는 값이다.

Clock/Timeout 모델은 “실제로 기다리지 않고도 장비 대기 시간 초과를 테스트하는 구조”다.

`ManualClock`을 쓰면 테스트에서 시간을 30초 앞으로 보내고, 상태머신이 Timeout을 감지해 `Alarmed`로 바뀌는지 확인할 수 있다.

## Goal 004: Clock / Timeout 모델

### 한 일

- `IClock` 인터페이스를 추가했다.
- 실제 시간용 `SystemClock`을 추가했다.
- 테스트/시뮬레이션용 `ManualClock`을 추가했다.
- 상태별 Timeout 설정을 위한 `StateTimeoutPolicy`를 추가했다.
- Timeout 검사 결과를 위한 `TimeoutCheckResult`를 추가했다.
- 상태머신에 `StateEnteredAtUtc`와 `CheckTimeout()`을 추가했다.
- Timeout 발생 시 `EquipmentEvent.Timeout` 이벤트로 `Alarmed` 상태 전이를 기록한다.

### 검증

- 로컬 빌드: 성공
- 로컬 테스트: 17개 통과
- GitHub Actions: PR 생성 후 확인 예정

### 사용자가 이해해야 할 것

Timeout은 장비가 특정 상태에서 너무 오래 머무를 때 알람으로 전환하는 기준이다.

실제 장비에서는 센서 입력이나 완료 신호가 정해진 시간 안에 들어오지 않으면 Timeout이 발생한다. 이번 구현은 그 개념을 장비 없이 테스트할 수 있게 만든 것이다.

### 한계와 다음 작업
