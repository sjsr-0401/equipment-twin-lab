## 소프트웨어 아키텍처 설명

```text
ClearAlarm command
    ↓
EquipmentCellController.CheckAlarmRecoveryCondition()
    ↓
Virtual IO 상태 확인
    ↓
허용: EquipmentStateMachine.Apply(ClearAlarm)
거부: EquipmentStateMachine.RejectCommand(ClearAlarm)

Scenario batch report
    ↓
ScenarioCliRun
    ↓
BuildMarkdownReport()
    ↓
DescribeActiveAlarm() / DescribeClearCondition()
```

복구 조건은 IO 상태를 확인해야 하므로 `EquipmentCellController`에 둔다.

CLI 리포트는 장비 로직을 새로 실행하지 않고 이미 실행된 `ScenarioRunner`의 최종 상태를 읽는다. 그래서 리포트 계층은 Core 로직을 바꾸지 않는 표시 계층이다.

## 추가 확인: Visual Studio launch profile 경로 수정

사용자가 Visual Studio build/debug 때 메시지가 떴다고 알려줘서 다시 확인했다.

확인 결과:

- `dotnet build EquipmentTwinLab.sln --configuration Debug` 성공
- 경고 0개
- 오류 0개
- `EquipmentTwin.Core.Tests` debug profile 실행 성공
- 기존 `EquipmentTwin.Cli` debug profile은 실행 위치에 따라 `Scenario file was not found` 또는 `Scenario path was not found`가 날 수 있었다.

수정한 것:

- `src/EquipmentTwin.Cli/Properties/launchSettings.json`에서 `workingDirectory`를 repo root로 맞췄다.
- CLI profile의 시나리오 경로를 repo root 기준으로 바꿨다.
- `docs/visual-studio.md`에 해당 오류의 의미와 해결 방법을 추가했다.

재검증:

- `CLI - normal cycle` 실행 성공
- `CLI - loading timeout` 실행 성공
- `CLI - batch report` 실행 성공, 7개 시나리오 통과

막힌 점:

- 현재 막힌 점 없음.

소프트웨어적으로 알아야 할 점:

Visual Studio에서 build가 성공해도 debug profile의 상대 경로가 틀리면 실행 시나리오 파일을 못 찾을 수 있다. 이건 C# 로직 오류가 아니라 실행 환경 설정 문제다. 그래서 launch profile에는 `workingDirectory`를 명시해두는 편이 유지보수에 안전하다.
